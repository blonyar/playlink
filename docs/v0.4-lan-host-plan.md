# Playlink v0.4 LAN Discovery & Host Mode Groundwork Plan

## 1. Stage Goal

v0.4 should make Playlink easier to use for local multiplayer sessions without turning the current WebSocket room server into a broad networking platform.

The goal is groundwork, not a full relay or P2P stack:

```text
Start server locally -> discover it on LAN or copy host address -> join rooms using the existing v0.3 protocol
```

v0.3 stabilized the room lifecycle and JSON protocol. v0.4 should build on that by making local hosting and local discovery explicit, inspectable, and modular.

## 2. Why This Stage Comes Next

The current server works when clients already know the WebSocket URL:

```text
ws://localhost:7777/ws
```

For LAN parties and small prototypes, manual URL entry is a usability blocker. Players need to answer:

- Which server should I connect to?
- What address should I share with friends on the same network?
- Is this process running as a dedicated server or a local host?
- Is LAN discovery enabled?
- What room/server metadata is safe to advertise?

v0.4 should answer those questions before adding relay or P2P behavior.

## 3. Scope

### In Scope

- LAN/host mode design document
- Server identity and host metadata model
- Debug/admin endpoint for server network/topology information
- Web Console display for local host information
- LAN discovery protocol design
- Optional minimal LAN discovery prototype if it can stay isolated
- Configuration names for host mode and discovery behavior
- Repeatable local checks for host metadata and discovery output

### Out of Scope

Do not implement these in v0.4:

- Relay mode
- P2P/NAT traversal
- STUN/TURN
- global matchmaking
- accounts/authentication
- production-grade admin permissions
- binary protocols
- SDK packages
- cross-internet server registry
- automatic firewall/router configuration

## 4. Current Baseline

Already available after v0.3:

- Rust Axum server
- WebSocket endpoint `/ws`
- HTTP health endpoint `/health`
- room list endpoint `/api/rooms`
- room detail endpoint `/api/rooms/:room_id`
- structured JSON protocol
- create/join/leave/list rooms
- room broadcasts
- player sessions
- ping/pong and idle timeout
- empty room cleanup
- Web Debug Console
- JS smoke and negative integration scripts
- Rust protocol and room tests

Current missing pieces for v0.4:

- no explicit server identity
- no `/api/server` or `/api/network` endpoint
- no advertised host display in Web Console
- no LAN discovery mechanism
- no host-mode configuration language
- no discovery tests/scripts

## 5. Proposed v0.4 User Flow

### 5.1 Dedicated Local Server

A developer starts Playlink:

```bash
rustup run stable cargo run
```

The console shows:

```text
Playlink server running
HTTP:      http://0.0.0.0:7777
WebSocket: ws://0.0.0.0:7777/ws
Mode:      dedicated
Discovery: disabled
```

The Web Console shows local connection information and the existing room controls.

### 5.2 Host Mode

A game or developer starts Playlink as a local host:

```bash
PLAYLINK_TOPOLOGY=host rustup run stable cargo run
```

Expected behavior:

- still uses the existing v0.3 room protocol
- still stores room state in memory
- exposes host metadata through admin/debug API
- does not add persistence, accounts, or matchmaking
- may eventually be embedded or launched by a game process

### 5.3 LAN Discovery

With LAN discovery enabled:

```bash
PLAYLINK_LAN_DISCOVERY=1 rustup run stable cargo run
```

Clients on the same network can discover the server and then connect using the normal WebSocket endpoint.

Discovery should find servers; it should not replace the room protocol.

## 6. Server Metadata Model

Add a small metadata model that can support host display and discovery.

Suggested fields:

```json
{
  "server_id": "uuid-or-stable-process-id",
  "name": "kang's Playlink Server",
  "version": "0.4.0-dev",
  "topology": "dedicated",
  "bind_addr": "0.0.0.0:7777",
  "websocket_path": "/ws",
  "http_url": "http://192.168.1.20:7777",
  "ws_url": "ws://192.168.1.20:7777/ws",
  "public_http_url": "http://192.168.1.20:7777",
  "public_ws_url": "ws://192.168.1.20:7777/ws",
  "discovery": {
    "enabled": false,
    "method": null,
    "port": 7778
  }
}
```

Initial topology values:

```text
dedicated
host
```

Reserved future values:

```text
relay
p2p
```

## 7. Admin/API Additions

### 7.1 Server Info Endpoint

Suggested endpoint:

```text
GET /api/server
```

Response:

```json
{
  "server_id": "playlink:Playlink Server:dedicated:0.0.0.0:7777",
  "name": "Playlink Server",
  "version": "0.4.0-dev",
  "topology": "dedicated",
  "bind_addr": "0.0.0.0:7777",
  "websocket_path": "/ws",
  "discovery": {
    "enabled": false,
    "method": null,
    "port": 7778
  }
}
```

This endpoint should be read-only and debug-oriented, like the current room inspection API.

### 7.2 Why Not Put This in `/health`?

`/health` should stay small and stable for process checks.

`/api/server` can include development/debug metadata that may grow over time.

## 8. LAN Discovery Design Options

### Option A: UDP Broadcast Query/Response

Client broadcasts a discovery query on a configured UDP port.

```json
{
  "type": "playlink_discovery_query",
  "version": 1
}
```

Server replies:

```json
{
  "type": "playlink_discovery_response",
  "version": 1,
  "server_id": "playlink:LAN Host:host:0.0.0.0:7777",
  "name": "LAN Host",
  "server_name": "LAN Host",
  "topology": "host",
  "http_port": 7777,
  "ws_path": "/ws",
  "public_http_url": "http://192.168.1.20:7777",
  "public_ws_url": "ws://192.168.1.20:7777/ws"
}
```

Pros:

- simple
- no external dependency
- easy to test with local scripts

Cons:

- broadcast behavior differs by OS/network
- may be blocked by firewall
- requires UDP socket handling

### Option B: mDNS

Advertise a service such as:

```text
_playlink._tcp.local
```

Pros:

- standard local discovery model
- good fit for LAN tools

Cons:

- dependency and platform behavior need evaluation
- more moving parts than UDP broadcast

### Option C: Manual Host Info Only

v0.4 could stop at `/api/server` and Web Console host display, leaving active discovery for v0.5.

Pros:

- very low risk
- immediately useful
- no UDP/firewall complexity

Cons:

- does not fully solve LAN discovery

## 9. Recommended v0.4 Strategy

Use a two-step v0.4:

### v0.4.1 Host Metadata

Implement:

- `ServerConfig` fields for topology and display name
- `/api/server`
- Web Console Network/Host display
- docs for manual LAN hosting
- tests for server info serialization/API response

### v0.4.2 LAN Discovery Prototype

Then add one isolated discovery mechanism, preferably UDP broadcast first, only if it does not complicate the room server.

Keep discovery behind config:

```text
PLAYLINK_LAN_DISCOVERY=1
PLAYLINK_DISCOVERY_PORT=7778
```

Discovery code should live in its own module, not inside room/session/websocket handling.

Suggested future file:

```text
src/discovery.rs
```

## 10. Configuration Names

Suggested environment variables:

```text
PLAYLINK_SERVER_ID=
PLAYLINK_SERVER_NAME=Playlink Server
PLAYLINK_TOPOLOGY=dedicated
PLAYLINK_PUBLIC_HTTP_URL=
PLAYLINK_PUBLIC_WS_URL=
PLAYLINK_LAN_DISCOVERY=0
PLAYLINK_DISCOVERY_PORT=7778
```

Rules:

- `PLAYLINK_TOPOLOGY` defaults to `dedicated`.
- `host` should not change room semantics yet; it only changes metadata/display.
- public URLs are optional overrides for display and discovery.
- discovery is disabled by default.

## 11. Web Console Additions

Add a small Network/Host section to the existing console before creating a larger admin UI.

Show:

```text
Server name
Topology
Bind address
Public HTTP URL when configured
WebSocket path or public WebSocket URL when configured
LAN discovery status
```

Do not add production admin controls yet.

Optional actions:

- copy WebSocket URL
- refresh server info

## 14. Risks

### OS and Firewall Differences

LAN discovery can behave differently across Windows, macOS, Linux, routers, and firewalls. Keep the first implementation optional and easy to disable.

### Scope Creep into Matchmaking

LAN discovery should only find reachable servers. It should not become global matchmaking, accounts, or room ranking.

### Coupling Discovery to Core Rooms

Discovery should advertise server metadata. It should not depend on room internals or mutate room state.

### Host Mode Becoming a New Backend

Host mode should initially be metadata and packaging behavior around the same room server. Avoid new authority models until there is a concrete game example.

## 15. Recommended Implementation Order

1. Add this v0.4 planning document.
2. Update README roadmap/current status to mention v0.3 completion and v0.4 direction.
3. Add server metadata config structs.
4. Add `GET /api/server`.
5. Add Web Console server/network display.
6. Add tests for metadata defaults and endpoint output.
7. Run existing checks.
8. Decide whether to continue into UDP discovery prototype or keep it as v0.4.2.

## 16. Explicit Non-Decision

Do not choose final P2P, relay, or NAT traversal strategy in v0.4. LAN discovery and host metadata should make local play easier while preserving the current v0.3 protocol as the stable base.
