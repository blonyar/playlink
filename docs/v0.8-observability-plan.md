# Playlink v0.8 Observability / Room Stats Plan

## 1. Stage Goal

v0.8 adds a real observability vertical slice to Playlink.

The goal is to make the current room server easier to inspect while preserving the small room-based multiplayer scope:

```text
room/server runtime state -> HTTP stats API -> Web Console dashboard -> JS verification -> docs
```

This is a framework capability, not a production metrics platform.

## 2. Why This Comes Next

v0.1-v0.7 established the room server, protocol, debug console, JavaScript helper, LAN/host groundwork, and relay planning.

The next useful capability is visibility:

- How long has the server been running?
- How many rooms and players are active?
- How many rooms have been created since startup?
- How many room messages have been broadcast?
- How old is a room?
- How active is a room?

This helps developers debug small multiplayer prototypes without adding databases, accounts, or production observability infrastructure.

## 3. Scope

### In Scope

- Add lightweight in-memory server stats.
- Add `GET /api/stats`.
- Track server uptime.
- Track active room count and player count.
- Track total rooms created since startup.
- Track total room messages broadcast since startup.
- Track per-room creation time.
- Track per-room message count.
- Display stats in the Web Debug Console.
- Verify stats from JavaScript smoke or SDK demo scripts.
- Document the endpoint and fields.

### Out of Scope

Do not implement these in v0.8:

- metrics database
- Prometheus/OpenTelemetry integration
- authentication or admin permissions
- persistent analytics
- historical charts
- billing/quotas
- MMO-scale telemetry
- global dashboards
- relay runtime
- P2P/NAT traversal

## 4. Proposed API

### `GET /api/stats`

Example response:

```json
{
  "uptime_seconds": 120,
  "room_count": 2,
  "player_count": 5,
  "total_rooms_created": 12,
  "total_messages_broadcast": 84
}
```

Field notes:

| Field | Meaning |
| --- | --- |
| `uptime_seconds` | Seconds since this Playlink process started. |
| `room_count` | Current active rooms. |
| `player_count` | Current active players across rooms. |
| `total_rooms_created` | Rooms created since process start. |
| `total_messages_broadcast` | Room messages broadcast since process start. |

### Room snapshots

Extend room list/detail data with:

```json
{
  "created_at_unix_secs": 1234567890,
  "message_count": 42
}
```

Field notes:

| Field | Meaning |
| --- | --- |
| `created_at_unix_secs` | Server-side Unix timestamp when room was created. |
| `message_count` | Number of `room_message` broadcasts accepted for this room. |

## 5. Architecture

Keep stats in memory and local to the running process.

Suggested ownership:

```text
Config/AppState
  -> started_at
RoomRegistry
  -> total_rooms_created
  -> total_messages_broadcast
  -> room snapshots with created_at/message_count
```

Rules:

- Do not add topology-specific logic to `RoomRegistry`.
- Do not add database persistence.
- Use atomics or existing room locks where appropriate.
- Keep stats approximate enough for debugging, but internally consistent for tests.

## 6. Web Console Display

Add stats to Overview:

- Uptime
- Total rooms created
- Total messages broadcast

Enhance room table/detail:

- room age or created timestamp
- room message count

Keep UI simple and readable. No charts yet.

## 7. JavaScript Verification

Update `examples/js-client/smoke.js` or `sdk-demo.js` to verify:

- `/api/stats` exists
- `room_count` and `player_count` reflect active state
- `total_rooms_created` is at least 1 after creating a room
- `total_messages_broadcast` increases after room messages
- room snapshot `message_count` increases after room messages

Prefer `smoke.js` because it already validates HTTP room state.

## 11. Risks

### Time handling in tests

Avoid brittle tests that require exact timestamps. Assert presence/ranges instead.

### Overbuilding metrics

Do not build charts, persistence, scraping endpoints, or production metrics in this stage.

### Stats consistency

Keep semantics simple: counts are per-process and reset on restart.

## 12. Future Extensions

Possible later additions:

- Web Console charts
