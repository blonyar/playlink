# Playlink v0.7 Relay Groundwork Plan

## 1. Stage Goal

v0.7 defines relay mode boundaries before any relay implementation begins.

The goal is groundwork, not production relay infrastructure:

```text
Existing room server -> explicit relay terminology -> topology/protocol impact review -> safe future implementation path
```

Relay should help small room-based games connect across networks when a direct dedicated/host connection is not practical. It must not turn Playlink into a global matchmaking platform, account system, MMO backend, or production infrastructure service.

## 2. Why This Comes After v0.6

v0.6 stabilized the JavaScript helper and documented the current client lifecycle. That makes relay planning safer because relay will affect how clients discover, choose, and connect to servers.

Before writing relay code, Playlink needs answers to:

- What does `relay` mean in this project?
- Is relay a topology, a deployment mode, or a separate service?
- Which protocol concepts stay unchanged?
- Which metadata should be advertised?
- What is explicitly out of scope?
- What can be tested without building production relay infrastructure?

## 3. Definitions

### Dedicated Server

A server process hosts rooms directly. Clients connect to the server's WebSocket endpoint.

```text
Client A ─┐
Client B ─┼── Dedicated Playlink Server
Client C ─┘
```

This is the current stable baseline.

### Host Mode

A player or game process runs a local Playlink server, and other players connect to it directly when reachable.

```text
Client B ─┐
Client C ─┼── Host player's Playlink Server
Client D ─┘
```

Current host mode is metadata/configuration groundwork, not a separate runtime architecture.

### Relay Mode

A relay helps clients exchange room traffic through an intermediate reachable service when direct connectivity is not practical.

Conceptually:

```text
Client A ─┐
Client B ─┼── Relay ── Room authority or host
Client C ─┘
```

For v0.7 planning, relay is a future topology option and design boundary. It is not implemented yet.

### P2P / NAT Traversal

P2P attempts direct client-to-client connectivity. NAT traversal may involve STUN/TURN-like mechanisms.

P2P and NAT traversal remain future experiments, not part of v0.7 implementation.

## 4. Scope

### In Scope

- Relay terminology and non-goals.
- Topology model review.
- Metadata fields that may be needed later.
- Protocol impact review.
- Candidate relay architectures.
- Safe implementation stages.
- Testing strategy for future relay work.
- Documentation updates only, unless a later v0.7 task explicitly chooses a tiny metadata-only code change.

### Out of Scope

Do not implement these in v0.7 groundwork:

- production relay server infrastructure
- global matchmaking
- accounts or authentication
- payments, billing, quotas, tenancy, or rate plans
- anti-cheat
- persistent room database
- NAT traversal implementation
- STUN/TURN integration
- automatic firewall/router configuration
- high-concurrency MMO-scale routing
- game-specific server simulation

## 5. Design Guardrails

Relay work must preserve these rules:

- The current room protocol remains usable over direct WebSocket.
- Relay must be optional.
- Relay must not become a required dependency for dedicated or LAN play.
- Relay planning should not add topology-specific logic to `RoomRegistry`.
- Room lifecycle semantics should remain stable: create, join, leave, room message, ping.
- Prefer additive metadata and new modules over changing existing payloads.
- Any future code change must be testable without a production relay deployment.

## 6. Topology Boundary

Current topology values:

```text
dedicated
host
```

Possible future topology values:

```text
relay
p2p
```

v0.7 should decide whether `relay` should be:

1. A `Topology` enum variant on the server.
2. A separate `RelayConfig` advertised by `/api/server`.
3. A future deployment role separate from the current room server.

Recommended initial direction:

```text
Keep relay as a planned topology/deployment concept in docs first.
Do not add a `Relay` enum variant until a concrete metadata or routing need exists.
```

Reason:

- Current runtime cannot behave as a relay yet.
- Advertising `topology: relay` before behavior exists would confuse clients.
- A future metadata-only change can be made atomically once fields are clear.

## 7. Protocol Impact Review

The current client protocol should remain stable:

```text
create_room
join_room
leave_room
room_message
ping
```

Current server messages should remain stable:

```text
room_created
room_joined
room_left
player_joined
player_left
room_broadcast
event_lagged
pong
error
```

Relay should not require changing room payloads in the first design iteration.

Potential future additions may include:

- connection hint metadata in `/api/server`
- relay endpoint metadata
- topology capability list
- client SDK helper for choosing an endpoint
- diagnostic errors for unavailable relay paths

Protocol compatibility rule:

```text
Prefer HTTP metadata and SDK connection options over changing room messages.
```

## 8. Candidate Relay Architectures

### Option A: Relay as WebSocket Forwarder

Clients connect to a relay WebSocket endpoint. The relay forwards messages to a room authority.

Pros:

- conceptually simple
- preserves JSON/WebSocket protocol
- easy to inspect

Cons:

- relay must manage upstream/downstream sessions
- error handling and backpressure become more complex
- authority boundary must be explicit

### Option B: Relay as Room Authority

The relay is also the room server. Clients connect to it directly.

Pros:

- close to current dedicated server behavior
- minimal protocol change

Cons:

- may just be a hosted dedicated server under a different name
- does not solve host-behind-NAT scenarios by itself

### Option C: Relay as Host Bridge

A host connects outbound to the relay; clients also connect to the relay. The relay bridges traffic between clients and host authority.

Pros:

- useful for hosts behind NAT
- closer to the long-term relay value

Cons:

- more complex session model
- host authority lifecycle needs careful design
- harder to test without more infrastructure

Recommended design sequence:

1. Document all options.
2. Start with metadata and terminology.
3. Prototype only after choosing authority boundaries.

## 9. Admin and Metadata Considerations

Current `/api/server` metadata includes:

- `server_id`
- `name`
- `version`
- `topology`
- `bind_addr`
- `websocket_path`
- `http_url`
- `ws_url`
- `public_http_url`
- `public_ws_url`
- `discovery`

Future relay metadata could be additive:

```json
{
  "relay": {
    "enabled": false,
    "mode": null,
    "public_ws_url": null
  }
}
```

Do not add this until a real client or prototype needs it.

## 10. SDK Considerations

The JavaScript helper should not hide relay behavior behind magic connection logic yet.

Future helper additions may include:

```js
const server = await client.serverInfo();
// choose direct URL, public URL, discovered URL, or relay URL explicitly
```

Rules:

- keep explicit `wsUrl` and `httpUrl` options
- document endpoint selection
- avoid automatic global discovery
- avoid account/session assumptions

Potential future tests:

## 13. Recommended Implementation Order

1. Add this planning document.
2. Link it from README and `docs/goal.md` in a separate documentation commit.
3. Decide whether relay needs a metadata-only design doc before code.
4. If needed, add `docs/relay-metadata.md` before changing Rust structs.
5. Only then consider a tiny metadata-only code change.
6. Do not build relay routing until authority boundaries are clear.

## 14. First Code Change Criteria

The first relay-related code change should happen only when all are true:

- there is a documented field or enum with a clear client use
- direct dedicated/host behavior remains unchanged
- tests can cover it locally
- README and protocol docs can explain it simply
- it can be reverted without affecting room core

Possible first code changes later:

- add a reserved `Relay` topology variant only when it can be advertised honestly
- add relay config parsing only when no runtime behavior is implied
- add relay metadata only when SDK endpoint selection uses it

## 15. Explicit Non-Decision

v0.7 does not choose final relay architecture. It creates the decision framework so later implementation can be small, modular, and testable.

Do not implement relay routing, NAT traversal, or matchmaking as part of this plan.
