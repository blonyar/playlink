# Playlink v0.1 Spec

## 1. Stage Goal

v0.1 should prove the smallest useful Playlink loop:

```text
Start server → connect clients → create room → join room → exchange messages → inspect server state
```

This stage is about making the dedicated WebSocket room server usable and testable. It should not expand into LAN discovery, relay, P2P, account systems, databases, or a full admin product.

## 2. Target User Story

A game developer can start Playlink locally, connect two test clients, create a room, join both clients into that room, send messages between them, and inspect active rooms through HTTP/debug tools.

## 3. Required Features

### Server Startup

- Server starts with one command.
- Server listens on `0.0.0.0:7777` by default.
- Server logs startup address.

Acceptance:

```bash
rustup run stable cargo run
curl http://localhost:7777/health
```

`/health` returns an OK JSON response.

### Health Endpoint

Endpoint:

```text
GET /health
```

Returns:

```json
{
  "status": "ok",
  "name": "playlink",
  "version": "0.1.0"
}
```

### Room List Endpoint

Endpoint:

```text
GET /api/rooms
```

Returns a JSON array of room snapshots:

```json
[
  {
    "id": "room uuid",
    "name": "test",
    "max_players": 4,
    "player_count": 2
  }
]
```

Acceptance:

- Empty server returns `[]`.
- After creating and joining a room, the room appears in the list.
- `player_count` reflects current connected players.

### WebSocket Endpoint

Endpoint:

```text
ws://localhost:7777/ws
```

The server accepts text JSON messages only for v0.1.

Invalid JSON or invalid message shapes should return a structured error message instead of crashing the connection handler.

### Create Room

Client message:

```json
{
  "type": "create_room",
  "payload": {
    "room_name": "test",
    "max_players": 4
  }
}
```

Server response:

```json
{
  "type": "room_created",
  "payload": {
    "room_id": "uuid"
  }
}
```

Acceptance:

- Room ID is generated by the server.
- Missing `room_name` uses a default room name.
- Missing `max_players` uses the default limit.
- Created room is visible from `/api/rooms`.

### Join Room

Client message:

```json
{
  "type": "join_room",
  "payload": {
    "room_id": "uuid",
    "player_name": "kang"
  }
}
```

Server direct response:

```json
{
  "type": "room_joined",
  "payload": {
    "room_id": "uuid",
    "player_id": "uuid"
  }
}
```

Room broadcast:

```json
{
  "type": "player_joined",
  "payload": {
    "player_id": "uuid",
    "player_name": "kang"
  }
}
```

Acceptance:

- Joining a missing room returns an error.
- Joining a full room returns an error.
- Joining a new room leaves the previous room first.
- Room player count updates.

### Leave Room

Client message:

```json
{
  "type": "leave_room"
}
```

Room broadcast:

```json
{
  "type": "player_left",
  "payload": {
    "player_id": "uuid"
  }
}
```

Acceptance:

- Player is removed from the room.
- Room player count updates.
- Calling leave while not in a room does not crash.

### Disconnect Cleanup

When a WebSocket disconnects:

- player is removed from the current room
- room receives `player_left`
- room event subscription task is stopped

Acceptance:

- Closing one test client decreases room player count.
- Other clients in the room receive a leave event.

### Room Message Broadcast

Client message:

```json
{
  "type": "room_message",
  "payload": {
    "data": {
      "x": 10,
      "y": 20
    }
  }
}
```

Room broadcast:

```json
{
  "type": "room_broadcast",
  "payload": {
    "from": "player uuid",
    "data": {
      "x": 10,
      "y": 20
    }
  }
}
```

Acceptance:

- Sender must be in a room.
- Message is broadcast to room subscribers.
- Payload is arbitrary JSON.

### Ping/Pong

Client message:

```json
{
  "type": "ping"
}
```

Server response:

```json
{
  "type": "pong"
}
```

Acceptance:

- Ping works before or after joining a room.
- Ping does not affect room state.

## 4. Test Client Requirement

v0.1 should include a simple test client before adding larger features.

Preferred first client:

```text
examples/js-client
```

Minimum capability:

- connect to `ws://localhost:7777/ws`
- create room
- join room
- send room message
- print received messages

A CLI client is enough. A polished browser UI is not required for v0.1.

## 5. Manual Smoke Test

A v0.1 implementation is acceptable when this flow works:

1. Start server.
2. Open client A.
3. Client A creates room.
4. Client A joins room.
5. Open client B.
6. Client B joins same room.
7. Client A sends message.
8. Client B receives broadcast.
9. Client B sends message.
10. Client A receives broadcast.
11. `GET /api/rooms` shows one room with two players.
12. Close client B.
13. Client A receives `player_left`.
14. `GET /api/rooms` shows one player.

## 6. Out of Scope for v0.1

Do not implement in this stage:

- user accounts
- authentication
- database persistence
- global matchmaking
- LAN discovery
- host mode
- relay mode
- P2P/NAT traversal
- binary protocol
- full Web Admin Console
- game-specific simulation logic
- anti-cheat
- deployment automation

## 7. Implementation Notes

Keep state in memory for v0.1.

Do not split into multiple crates yet unless the single-crate structure becomes painful. The current single-crate layout is easier to change while the protocol and room model are still forming.

Prefer validating the WebSocket room loop with a test client before adding new modules.
