# Playlink v0.9 Lightweight State Sync Prototype Plan

## 1. Stage Goal

v0.9 should add a small, example-first state sync layer without changing the stable room protocol.

The goal is to help simple 2D multiplayer prototypes share lightweight state snapshots using the existing `room_message` broadcast path:

```text
client state -> SDK helper/convention -> room_message -> room_broadcast -> interpolation/example UI
```

This is not a server-authoritative simulation layer.

## 2. Why This Comes Next

v0.8 made the room server easier to inspect. The next useful capability is a clearer sync convention for games that need more than isolated events but less than a full game networking stack.

The browser mini-game already sends movement messages. v0.9 should make that pattern more explicit, documented, and reusable.

## 3. Scope

### In Scope

- Document a lightweight state snapshot convention inside `room_message.payload.data`.
- Add small JavaScript helper utilities for throttled state publishing if they stay simple.
- Update or extend the browser mini-game to demonstrate structured state snapshots.
- Keep interpolation client-side and example-owned.
- Add repeatable JavaScript checks for deterministic helper behavior when practical.
- Update docs so event sync and state sync are clearly different choices.

### Out of Scope

- New core WebSocket message types.
- Server-authoritative gameplay state.
- Physics replication.
- rollback netcode.
- ECS/world replication.
- binary protocol changes.
- persistence, matchmaking, accounts, or anti-cheat.

## 4. Proposed Convention

Prefer additive conventions inside room message data:

```json
{
  "kind": "state_snapshot",
  "tick": 123,
  "entity_id": "player:alice",
  "state": {
    "x": 54,
    "y": 50,
    "vx": 1,
    "vy": 0
  }
}
```

Field notes:

| Field | Meaning |
| --- | --- |
| `kind` | Convention discriminator. Use `state_snapshot` for snapshot-style updates. |
| `tick` | Client-side monotonic sequence number. Used for ordering and stale update rejection. |
| `entity_id` | Stable entity key meaningful to the example/game. |
| `state` | JSON object owned by the game/example. |

The server should treat this as ordinary room message data.

## 5. Architecture Rules

- Keep `RoomRegistry` unaware of state sync semantics.
- Keep protocol compatibility: raw `room_message` remains the transport.
- Put helper logic in JavaScript examples or SDK helper utilities first.
- Keep Web Console useful by displaying raw messages without trying to become a game-state debugger.
- Avoid introducing a generic replication engine until real examples justify it.

## 6. Candidate Implementation Sequence

1. Add this v0.9 plan.
2. Update `docs/sync-models.md` to name v0.9 as the next sync milestone.
3. Add a tiny JS helper for throttled snapshot publishing or stale tick filtering.
4. Add syntax/unit-style checks for the helper if it is factored into reusable code.
5. Update the mini-game example to use `state_snapshot` or add a clear state-sync mode.
6. Update `docs/js-client-api.md`, `docs/demo-guide.md`, and README examples.
7. Run `scripts/verify.ps1` and manually verify the mini-game in a browser.

## 7. Acceptance Criteria

v0.9 is complete when:

- [x] `docs/v0.9-state-sync-prototype-plan.md` exists.
- [x] `docs/sync-models.md` documents v0.9 state sync conventions.
- [x] The JavaScript example path demonstrates `state_snapshot` messages.
- [x] Any reusable helper has focused checks or is covered by an existing integration script.
- [x] README and demo docs explain when to use event sync vs state sync.
- [x] `scripts/verify.ps1` passes.
- [x] Browser mini-game state-sync behavior is manually verified.

## 8. Risks

### Over-generalizing too early

Keep v0.9 example-led. Do not design a full replication framework from one mini-game.

### Protocol churn

Do not add new protocol messages until the room-message convention proves insufficient.

### Client authority confusion

Document clearly that this prototype is client-published state, not authoritative validation.
