# Playlink v0.3 Protocol & Room Reliability Plan

## 1. Stage Goal

v0.3 focuses on making the current WebSocket room server more reliable, predictable, and ready for SDKs or LAN discovery.

The goal is not to add a new networking mode yet. The goal is to stabilize the protocol and room lifecycle so future clients can depend on it.

Current working loop:

```text
Connect client → create/join room → send messages → inspect rooms/players in Web Console
```

v0.3 should make that loop clearer and safer.

## 2. Why This Stage Comes Before LAN Discovery

LAN discovery, host mode, relay, and P2P will all depend on the same room lifecycle and protocol behavior.

If the current protocol is vague, every later client will inherit those problems.

So v0.3 should settle questions like:

- What error shape does the server return?
- What happens when a room is full?
- What happens when a player leaves?
- Can the UI explicitly leave a room?
- Can clients understand why an action failed?
- Which messages are stable enough for SDKs?

## 3. Scope

### In Scope

- Standard protocol error shape
- Basic error codes
- Explicit leave room flow in the Web Console
- Better room lifecycle events
- Better client state handling after leave/disconnect
- Room capacity behavior and UI feedback
- More predictable room cleanup
- Protocol documentation
- Basic integration tests or repeatable test scripts

### Out of Scope

Do not implement these in v0.3:

- LAN discovery
- relay mode
- P2P/NAT traversal
- accounts/authentication
- database persistence
- binary protocols
- SDK packages
- production admin permissions
- matchmaking

## 5. Protocol Improvements

### 5.1 Standard Error Response

Current error response:

```json
{
  "type": "error",
  "payload": {
    "message": "room not found"
  }
}
```

v0.3 should change or extend it to:

```json
{
  "type": "error",
  "payload": {
    "code": "room_not_found",
    "message": "Room not found"
  }
}
```

Suggested initial error codes:

```text
invalid_message
room_not_found
room_full
not_in_room
invalid_room_id
internal_error
```

### 5.2 Protocol Documentation

Add:

```text
docs/protocol.md
```

It should document:

- client messages
- server messages
- payload shapes
- error codes
- room lifecycle expectations
- examples

### 5.3 Stable Message Names

Review current message names before SDKs depend on them:

```text
ClientMessage:
- create_room
- join_room
- leave_room
- room_message
- ping

ServerMessage:
- error
- room_created
- room_joined
- player_joined
- player_left
- room_broadcast
- pong
```

Decide whether these names are acceptable for v0.3 stability.

## 6. Room Lifecycle Improvements

### 6.1 Explicit Leave Room UI

The protocol already has `leave_room`, but the Web Console does not expose it clearly.

Add a Leave Room button in one of these places:

- Simulator session state
- Messages current room area
- Room Detail panel when selected room is current

Expected behavior:

```text
Click Leave Room → send leave_room → clear current room state → refresh rooms → disable message send
```

### 6.2 Room Cleanup Behavior

Current behavior:

```text
When the last player leaves, room is deleted.
```

v0.3 should document this explicitly and ensure UI handles it gracefully:

- selected room detail clears if room no longer exists
- current room clears after leave/disconnect
- room list refreshes after cleanup

### 6.3 Join Failure UI

When joining fails, the console should show the returned error clearly.

Examples:

```text
room_not_found
room_full
invalid_room_id
```

- [ ] changes are pushed to origin

## 10. Risks

### Protocol churn

Changing protocol after SDKs exist will be more painful. v0.3 is the right time to settle error shape and message names.

### UI hiding server behavior

The Web Console should show server errors directly instead of hiding them behind generic UI messages.

### Overbuilding

Do not add accounts, persistence, matchmaking, or LAN discovery in this stage. Keep the focus on reliability and protocol clarity.

## 11. Recommended Implementation Order

1. Add `docs/protocol.md` with current protocol.
2. Introduce structured error codes in Rust protocol types.
3. Update WebSocket handlers to return error codes.
4. Update Web Console error display if needed.
5. Add Leave Room button and state cleanup.
6. Add negative JS test script.
7. Run all checks.
8. Commit and push.
