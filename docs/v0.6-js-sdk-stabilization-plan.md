# Playlink v0.6 JS SDK Stabilization Plan

## 1. Stage Goal

v0.6 stabilizes the JavaScript helper introduced in v0.5 so small game prototypes can depend on a clearer client lifecycle without requiring package publishing or a new backend feature.

The target loop is:

```text
Start server -> import PlaylinkClient -> connect -> create/join/leave rooms -> send game events -> handle events/errors predictably
```

This stage should keep the server room protocol stable while improving client ergonomics, documentation, and repeatable checks.

## 2. Why This Comes Before Relay/P2P

Relay and P2P will make client lifecycle more complex. Before adding those topology choices, Playlink should have a small but well-documented client API for:

- connection setup
- request/response correlation
- room lifecycle state
- event handling
- error handling
- close/timeout behavior

A stable JavaScript helper also makes future examples and integration tests cheaper to write.

## 3. Scope

### In Scope

- Document the current `PlaylinkClient` API.
- Add or refine helper methods only when they simplify the existing room loop.
- Ensure `leaveRoom()` uses the `room_left` acknowledgement.
- Add integration coverage or demo assertions for `room_left`.
- Document Node.js and browser runtime assumptions.
- Improve SDK demo readability and reliability.
- Keep examples small, copy-pasteable, and protocol-transparent.

### Out of Scope

Do not implement these in v0.6:

- npm package publishing
- authentication or accounts
- global matchmaking
- relay mode
- P2P/NAT traversal
- engine-specific SDKs
- binary protocol encoding
- production reconnection/session-resume semantics

## 5. Proposed API Surface

Keep `PlaylinkClient` small and explicit.

Constructor options:

```js
new PlaylinkClient({
  name,
  wsUrl,
  httpUrl,
  log,
  keepaliveIntervalMs,
})
```

Core methods:

```js
await client.connect()
client.close()
client.on(type, handler)
await client.createRoom({ roomName, maxPlayers })
await client.joinRoom(roomId, playerName)
await client.leaveRoom()
client.sendRoomMessage(data)
await client.ping()
await client.listRooms()
await client.serverInfo()
await client.request(type, payload, timeoutMs)
client.send(message)
await client.waitFor(type, predicate, timeoutMs)
```

State fields exposed for examples:

```js
client.socket
client.playerId
client.roomId
client.messages
```

These fields are acceptable for examples, but future packaged SDKs may hide or formalize them.

## 6. Behavior Requirements

### Connection

- `connect()` resolves after WebSocket open.
- `connect()` rejects on timeout or connection error.
- `close()` stops keepalive and closes the socket.
- pending requests reject when the socket closes.

### Requests

- `request()` sends an `id` field.
- server responses with matching `id` resolve or reject the pending request.
- `error` responses reject with `error.code`.
- timeouts remove pending requests.

### Room Lifecycle

- `createRoom()` returns `room_id`.
- `joinRoom()` sets `roomId` and `playerId` from `room_joined`.
- `leaveRoom()` waits for `room_left`, then clears `roomId`.
- `room_message` sends arbitrary JSON through the existing protocol.

### Events

- `on(type, handler)` registers event handlers and returns an unsubscribe function.
- `waitFor(type, predicate, timeoutMs)` checks existing message history before waiting.
- message history remains bounded.

## 9. Risks

### Over-packaging too early

Publishing an npm package before the API is stable would make protocol churn more expensive. Keep v0.6 example-oriented.

### Hiding protocol behavior

The helper should reduce boilerplate but still make protocol messages understandable. Examples should remain useful for developers who choose raw WebSocket.

### Reconnection scope creep

Automatic reconnect and session resume are valuable but can become complex. v0.6 may document close behavior and reject pending requests without promising full resume semantics.
