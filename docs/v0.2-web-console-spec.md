# Playlink v0.2 Web Debug Console Spec

## 1. Stage Goal

v0.2 adds a browser-based debug console served by the Playlink server.

The console is not a production admin panel. It is a developer tool for seeing server state and manually exercising the current WebSocket room flow.

The smallest useful loop is:

```text
Start server → open browser → see health/rooms → connect test client → create/join/send → watch messages
```

## 2. UI Strategy

Use plain HTML, CSS, and JavaScript for v0.2.

Do not introduce React, Vite, Tauri, Electron, or a desktop UI yet. The server should serve the console directly so it works on Linux, Windows, macOS, and other LAN devices through a browser.

## 3. Server Routes

Add static console routes:

```text
GET /                    web console HTML
GET /assets/style.css    console CSS
GET /assets/app.js       console JS
```

Existing routes remain:

```text
GET /health
GET /api/rooms
GET /ws
```

## 4. Console Layout

The first UI should fit on one page.

Sections:

```text
Overview
Rooms
Test Client
Messages
```

### Overview

Show:

- health status
- server name
- server version
- room count
- total player count
- WebSocket connection state

### Rooms

Show a table of room snapshots:

- room ID
- name
- player count
- max players

Actions:

- refresh rooms
- copy/select room ID for joining

### Test Client

Controls:

- WebSocket URL input, defaulting to current host `/ws`
- connect/disconnect button
- room name input
- max players input
- create room button
- room ID input
- player name input
- join room button
- message payload textarea
- send room message button
- ping button

Behavior:

- creating a room should fill the room ID input
- joining uses the current room ID input
- received messages appear in the message log
- room list refreshes after important actions

### Messages

Show a chronological log of:

- WebSocket connection events
- sent messages
- received messages
- errors

JSON payloads should remain visible for debugging.

## 5. Acceptance Criteria

A v0.2 implementation is acceptable when:

1. `rustup run stable cargo run` starts the server.
2. Opening `http://localhost:7777/` shows the console.
3. The console loads CSS and JS from `/assets/...`.
4. Overview displays `/health` data.
5. Rooms section displays `/api/rooms` data.
6. The console can connect to `/ws`.
7. The console can create a room.
8. The created room appears in the room list.
9. The console can join that room.
10. The console can send a room message.
11. Received messages appear in the message log.
12. `cargo check` still passes.

## 6. Out of Scope

Do not implement in v0.2:

- authentication
- authorization
- production admin permissions
- charts
- persistent logs
- React/Vite build pipeline
- desktop wrapper
- room deletion UI unless the backend already supports it
- complex styling system
